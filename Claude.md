## 议题驱动开发

所有代码变更必须通过 GitHub Issue 进行追踪，无论是修改还是新增功能。

### 工作流程

1. **创建 Issue** - 在开始任何工作之前，先创建对应的 Issue
2. **讨论方案** - 在 Issue 中讨论实现方案，获得确认
3. **实现代码** - 按照 Issue 中确认的方案进行开发
4. **关联提交** - 提交时引用 Issue 编号（如 `fix #123` 或 `closes #123`）
5. **关闭 Issue** - 代码合并后自动或手动关闭 Issue

### Issue 类型

- **Bug 修复**: 描述问题现象、复现步骤、期望行为
- **新功能**: 描述需求背景、功能说明、验收标准
- **重构优化**: 说明当前问题、优化目标、影响范围
- **文档更新**: 说明需要更新的内容和原因

### Issue 标签

- `bug` - 缺陷修复
- `feature` - 新功能
- `enhancement` - 功能增强
- `refactor` - 代码重构
- `ai-safe` - AI 可安全自动处理
- `auto-fix-approved` - 已批准自动修复

### 注意事项

- 不要直接修改代码而不创建 Issue
- 一个 Issue 专注于一个问题或功能
- Issue 描述要清晰，便于他人理解和 AI 处理
- 复杂功能可拆分为多个子 Issue
